# Secure Password Database

## Challenge Information

* **Challenge Name:** Secure Password Database
* **Category:** Reverse Engineering
* **Difficulty:** Medium
* **Points:** 200

---

# Objective

Analyze the provided executable (`system.out`) to determine how the password verification works and recover the correct hash required to obtain the flag.

---

# Files Provided

* `system.out`

# Initial Analysis

The challenge provided an ELF executable named `system.out`.

First, the binary was downloaded:

```bash
cd /tmp
wget https://challenge-files.picoctf.net/.../system.out
chmod +x system.out
```

Running the challenge displayed the following prompts:

```text
Please set a password for your account:
How many bytes in length is your password?
You entered:
Your successfully stored password:
Enter your hash to access your account!
```

The program prints every byte of the entered password, indicating that it stores the password as raw byte values before requesting a hash.

---

# Reverse Engineering the Binary

The executable was opened in **Ghidra** for static analysis.

The decompiled source revealed the following hash function:

```c
long hash(byte *input)
{
    long hash = 0x1505;

    while (*input != 0) {
        hash = hash * 33 + *input;
        input++;
    }

    return hash;
}
```

This is the well-known **DJB2 hashing algorithm**.

Another important function discovered was:

```c
void make_secret(long ptr)
{
    for (i = 0; obf_bytes[i] != '\0'; i++) {
        *(ptr + i) = obf_bytes[i] ^ 0xAA;
    }

    *(ptr + 12) = '\0';

    hash(ptr);
}
```

From this function it became clear that:

* The program contains an XOR-obfuscated secret.
* Each byte is XORed with `0xAA` to reconstruct the original password.
* The reconstructed password is then passed to the `hash()` function.

This confirmed that the program internally computes a DJB2 hash of the secret password.

---

# Dynamic Analysis with GDB

To obtain the exact hash generated by the binary, **GDB** was used.

Launch GDB:

```bash
gdb ./system.out
```

Set a breakpoint on the hashing function:

```gdb
break hash
run
```

Provide sample input when prompted:

```text
Password: asdf
Length: 4
Hash: 23
```

Execution stops inside the `hash()` function.

Continue until the function returns:

```gdb
finish
```

Inspect the registers:

```gdb
info registers
```

The return value of the hash function is stored in the **RAX** register.

Example:

```text
RAX = -3209081493549540382
```

This value is the internally computed DJB2 hash of the reconstructed secret password.

# Key Takeaways

* Reverse engineering can reveal hidden verification logic even without source code.
* Ghidra identified the hashing algorithm as **DJB2**.
* The binary reconstructs the secret password using an XOR (`0xAA`) operation before hashing it.
* GDB allowed inspection of the function's return value in the **RAX** register, eliminating the need to manually reverse the password.
* The challenge demonstrates why client-side validation and exposing internal authentication logic are insecure.
